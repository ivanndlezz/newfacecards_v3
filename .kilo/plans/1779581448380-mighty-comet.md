# Plan: Modern Vertical Swipe Gesture for Top Menu Toggle

## Goal
Implement a modern, cross-device (mobile + desktop) vertical swipe/drag gesture detector in `admin/user/ivangonzalez/index.html`.

- **Swipe / drag upward** (negative deltaY) → set `data-top-menu="hidden"` on `#data-status-storage`
- **Swipe / drag downward** (positive deltaY) → set `data-top-menu="showing"` on `#data-status-storage`
- Must feel native, reliable, and not interfere with existing UI elements (buttons, island menu, bottom controls, drag handles).

## Current State Analysis

### File under edit
- `admin/user/ivangonzalez/index.html` (783 lines, the admin SPA binder for the user's card profile)

### Relevant DOM
- `#data-status-storage` (line 547) — currently only carries `data-bind-attr="config.cover_active"`. This is the canonical element for global UI state flags.
- `.main-content.main-content--profile` (line 595) — stable container around the card preview.
- `#card` (line 604) — content area that gets dynamically hydrated by the binder (line 658 loads card templates).
- Header (`.header`) with back button, publish, and a visual drag handle (`.handler--drag`).
- Fixed bottom sheet (`.controls`) with its own drag handle.
- `#island` (island-menu component) — positioned outside the flow.

### Existing gesture / input handling
- Zero touch, pointer, or swipe code in the entire project (confirmed via broad grep).
- Only standard `click` and custom island events (`island-action`, `island-mode-change`, etc.).
- Several `cursor: pointer` and one visual drag affordance, but no actual drag logic.

### No conflicting libraries
- Pure vanilla + ES modules. No Hammer, no gesture polyfills.

### Styling note
- No existing CSS rules target `[data-top-menu]`. The attribute is new; later CSS or JS can react to it (e.g., `header { transform: translateY(...) }` when `data-top-menu="hidden"`).

## Proposed Solution

### Technology choice: Pointer Events API (unified)
- Single modern API (`pointerdown`, `pointermove`, `pointerup`, `pointercancel`).
- Works for:
  - Touch (mobile)
  - Mouse drag (desktop)
  - Pen / stylus
- Better than separate `touch*` + `mouse*` handlers.
- Use `setPointerCapture()` for reliable tracking even if pointer leaves the element.
- Respects `touch-action` CSS if we want to fine-tune later.

### Architecture
Lightweight, self-contained vertical swipe detector:
1. Attach to a **stable container** that represents the "content area" users swipe on (`.main-content` recommended).
2. On `pointerdown`:
   - Record start coordinates + pointerId.
   - Ignore if the pointer started on interactive elements (buttons, links, inputs, island, controls, explicit drag handles).
   - Call `setPointerCapture`.
3. On `pointermove`:
   - Compute `deltaX`, `deltaY`.
   - Early exit if horizontal movement dominates (to not fight scrolling).
   - Optionally `preventDefault()` once vertical intent is clear (prevents page scroll jitter on touch).
4. On `pointerup` / `pointercancel`:
   - Calculate final vertical displacement.
   - If `|deltaY| > threshold`:
     - `deltaY < 0` (finger moved up) → `data-top-menu="hidden"`
     - `deltaY > 0` (finger moved down) → `data-top-menu="showing"`
   - Always release capture.
5. Cleanup on page unload (optional; listeners on long-lived elements are fine).

### Recommended attachment point
- **Target**: `.main-content` (the rounded card container)
  - Stable (not replaced by binder).
  - Large hit area matching the visual profile card.
  - Users naturally swipe here to "peek" or hide the top chrome.
- Alternative considered: `#app` (broader) or `body`. `.main-content` is the sweet spot.

### Exclusion / guard rules (critical for UX)
Ignore gesture start if `event.target.closest(...)` matches:
- `button, a, input, textarea, select`
- `[data-role]` (island buttons use this)
- `.controls, .handler, #island, .header__cta, .header__back-btn`

This prevents the swipe from stealing taps on Publish, Back, Island satellites, bottom sheet handle, etc.

### Thresholds & tuning (start values)
- `VERTICAL_THRESHOLD = 70` (pixels) — good balance between accidental and intentional.
- Horizontal tolerance: if `Math.abs(deltaX) > Math.abs(deltaY) * 1.2` during move, treat as horizontal scroll and abort.
- Optional future: velocity-based "flick" detection (time + distance), but distance-only is sufficient and simpler.

### Behavior on attribute
- **Always overwrite** the attribute (no toggle logic).
- The element can carry multiple data attrs; this is orthogonal to the existing `data-bind-attr`.
- Optionally dispatch a lightweight custom event (`top-menu-state-change`) for future listeners.

### Code placement
Add one new `<script>` block just before `</body>` (after the island bootstrap script, around line 781).

Keep it non-module (plain script) for simplicity and to avoid extra import.

Make the detector a tiny IIFE or named function for clarity and easy future extraction.

### Future extraction path (nice-to-have)
If the pattern proves useful elsewhere, move the detector to:
`shared/utilities/vertical-swipe-detector.js`
Export a function `initVerticalSwipe(target, { onUp, onDown, threshold })`.

For **this task** we keep it local to the admin page.

## Implementation Outline (step-by-step)

1. **Insert new script** at the end of `<body>` (before `</body>`).

2. **Inside the script**:
   ```js
   (function initTopMenuSwipe() {
     const storage = document.getElementById('data-status-storage');
     const surface = document.querySelector('.main-content');
     if (!storage || !surface) return;

     const THRESHOLD = 70;
     let startX = 0, startY = 0, pointerId = null;

     function isInteractive(el) {
       return !!el.closest('button, a, input, textarea, select, [data-role], .controls, .handler, #island, .header__cta, .header__back-btn');
     }

     function setTopMenuState(state) {
       storage.setAttribute('data-top-menu', state); // "hidden" | "showing"
       // Optional: storage.dispatchEvent(new CustomEvent('top-menu-state-change', { detail: { state } }));
     }

     surface.addEventListener('pointerdown', (e) => {
       if (isInteractive(e.target)) return;
       pointerId = e.pointerId;
       startX = e.clientX;
       startY = e.clientY;
       surface.setPointerCapture(pointerId);
     }, { passive: true });

     surface.addEventListener('pointermove', (e) => {
       if (e.pointerId !== pointerId) return;

       const dx = e.clientX - startX;
       const dy = e.clientY - startY;

       // Strong horizontal intent → let the browser scroll horizontally / swipe cards
       if (Math.abs(dx) > Math.abs(dy) * 1.2) {
         surface.releasePointerCapture(pointerId);
         pointerId = null;
         return;
       }

       // Once we have clear vertical movement we can claim the gesture
       if (Math.abs(dy) > 10) {
         e.preventDefault(); // reduce scroll conflict on touch
       }
     }, { passive: false });

     surface.addEventListener('pointerup', (e) => {
       if (e.pointerId !== pointerId) return;

       const dy = e.clientY - startY;
       surface.releasePointerCapture(pointerId);
       pointerId = null;

       if (Math.abs(dy) < THRESHOLD) return;

       if (dy < 0) {
         setTopMenuState('hidden');   // swiped up
       } else {
         setTopMenuState('showing');  // swiped down
       }
     });

     surface.addEventListener('pointercancel', () => {
       if (pointerId !== null) {
         surface.releasePointerCapture(pointerId);
         pointerId = null;
       }
     });
   })();
   ```

3. **Add minimal defensive CSS** (optional but recommended):
   ```css
   .main-content {
     touch-action: pan-y; /* allow vertical scroll, we only steal when clearly a menu swipe */
   }
   ```
   Place this inside the existing `<style>` block near the other `.main-content` rules.

4. **No changes** to HTML structure or binder logic required.

### Edge cases to verify
- Starting drag on Publish button, Back button, Island edit/satellites, bottom sheet handle → gesture ignored.
- Very small movements (< 70 px) → no state change.
- Diagonal drags → horizontal bias wins, gesture aborts.
- Rapid successive swipes → each one evaluated independently.

## Non-Goals (out of scope for this task)
- Visual animation of the top menu (that belongs in CSS reacting to the data attr).
- Keyboard support (arrow keys) for the gesture.
- Horizontal swipe detection.
- Velocity / momentum flicking (can be added later).
- Haptic feedback.
- Wheel / trackpad scroll-to-toggle (user asked for "deslizar").

## Risks & Mitigations
- **Conflict with native scrolling**: Mitigated by horizontal-bias abort + `touch-action: pan-y` + `preventDefault` only on clear vertical intent.
- **Interference with island / bottom sheet**: Strong `isInteractive` guard + exclusion selectors.
- **Multiple pointers**: Single `pointerId` tracking prevents race conditions.
- **Dynamic content inside #card**: Attachment on `.main-content` (ancestor) survives template replacement.

## Feature Extension: Scroll-Depth Driven Header Compression Effect

### Goal
Add a smooth, continuous visual effect to `.header__actions` that reacts to vertical scroll depth:

- At scroll position `0` → full size, no blur (`scale(1)`, `blur(0px)`)
- At scroll position `N` (and beyond) → compressed state (`scale(0.7)`, `blur(7px)`)

The effect must be driven by JavaScript calculating a normalized "scroll depth" value and applying it through CSS custom properties:

```css
.header__actions {
  transform: scale(var(--scale, 1));
  filter: blur(var(--blur, 0px));
  transition: all 0.3s ease;   /* already present */
}
```

This creates a nice "receding header" polish while the user scrolls into the card content.

### Current State (after previous work)
- The page now has a minimal click-to-toggle on `.handler` that sets `data-top-menu="hidden"` / `"showing"` on `#data-status-storage`.
- `.header__actions` is already `position: sticky; top: 30px;` with a 0.3s transition.
- `data-top-menu` currently only rotates the drag handle lines.
- No scroll listeners or dynamic CSS variable logic exists yet.
- The header sits above the large `.main-content` card area.

### Proposed Architecture

1. **CSS Variables (authoritative source of truth for the effect)**
   - `--scale`: number (1 → 0.7)
   - `--blur`: length (0px → 7px)
   - Set directly on the `.header__actions` DOM element (scoped, no global pollution).

2. **Scroll Calculator (pure function)**
   - Listen to `scroll` on `window` (or the nearest scrollable ancestor).
   - Compute `progress = clamp(0, window.scrollY / MAX_SCROLL_DEPTH, 1)`
   - Linear map:
     - `scale = (1 - 0.3 * progress).toFixed(3)`
     - `blur  = (7 * progress).toFixed(1) + 'px'`
   - Apply via `element.style.setProperty('--scale', scale)` and same for `--blur`.

3. **Performance**
   - Use a classic `ticking` + `requestAnimationFrame` pattern (or the modern `scroll` + rAF).
   - Passive listener.
   - Initialize values on DOMContentLoaded / script execution (handles deep-link or restored scroll position).

4. **Parameters (easy to tune)**
   ```js
   const MAX_SCROLL_DEPTH = 220;   // pixels until full effect (N)
   const MIN_SCALE = 0.7;
   const MAX_BLUR = 7;
   ```

5. **Interaction with existing `data-top-menu` toggle**
   Recommended behavior (to be confirmed):
   - Scroll always drives the continuous effect (independent of the boolean flag).
   - Clicking the handler still toggles the attribute (for the handle rotation and any future CSS that keys off `[data-top-menu="hidden"]`).
   - Optional enhancement: when `data-top-menu="hidden"` we can force the effect to max values (scale 0.7 / blur 7px) so the header looks fully collapsed even if the user scrolls back up. This gives a "locked collapsed" mode.

   Alternative (simpler): keep scroll effect always active; the click toggle remains purely for the data attribute.

### Implementation Outline

1. **CSS update** (one small addition inside the existing `.header__actions` rule):
   ```css
   .header__actions {
     ...
     transform: scale(var(--scale, 1));
     filter: blur(var(--blur, 0px));
   }
   ```

2. **JavaScript** (add inside the existing small IIFE at the bottom of the file, or as a sibling `<script>`):
   - Grab `const actions = document.querySelector('.header__actions');`
   - Define the constants.
   - Create `updateHeaderFromScroll()` function.
   - Attach `window.addEventListener('scroll', onScroll, { passive: true });`
   - Call once on init to set initial state.
   - Clean rAF ticking guard.

3. **Optional integration** with the current `setTopMenuState`:
   - After setting the attribute, we can also call a `forceHeaderEffect(state === 'hidden' ? 1 : null)` helper that either locks the values or lets scroll control resume.

4. **No changes** to the binder, island menu, or card template loading.

- Do we want the effect to respect the `data-top-menu` state (e.g. only apply scroll compression when the menu is "showing")?
- Should the effect also be applied when the page is loaded with the menu already in "hidden" state via other means?

### Testing Strategy
- Scroll slowly from top of the admin page and observe smooth scale + blur on the entire header actions row.
- Scroll back to top → values return to 1 / 0px.

### Risks & Mitigations
- Over-blurring can make text unreadable → 7px is the outer limit; we stop there.
- Sticky header + transform can sometimes create new stacking contexts → already using `transition` and will test z-index if needed.
- Scroll performance on low-end devices → rAF + passive listener + minimal DOM writes (only two style properties).

### Summary for this extension

- One CSS rule addition.
- Purely additive, zero dependencies.
