# Plan: Implement Card Theme Carousel (Visual Apariencia Selector)

## Goal
Realize the declared intention of `card-carousel.html`: a visual, interactive carousel of card theme previews.

When the user activates **"Apariencia"** mode via the island-menu:
- A horizontal scrollable / swipeable strip of theme preview cards appears.
- One bullet per theme is rendered in `.bullet-active-indicator`.
- The currently selected theme is reflected via `data-card-selected` on the container.
- Clicking a preview (or its bullet) selects it, updates the attribute, highlights the correct bullet, and **swaps the live card preview** using the existing binder + `loadTemplateAndRun` mechanism.

This replaces the current hardcoded `const themeName = "aurora"` in the admin SPA.

---

## Current State Analysis

### Declared Intent (card-carousel.html)
- Root: `#cards[data-card-selected="..."]`
- Children: multiple `.card[data-name="theme-slug"]`
- Sibling: `.bullet-active-indicator` (comment says: "for each card map bullets set active to actual active")

### Existing Architecture (already solid)
- **DataBinder + DomHydrator** (`shared/utilities/binder.js`) — handles `data-bind*` attributes and template loading via `loadTemplateAndRun`.
- **Card templates** — 6 production themes in `templates/cards/` (aurora, zen, tropico, obsidiana, brutal, deco). Each is a self-contained `<style>` + `<div class="card--xxx">` with data-bind hooks.
- **Island Menu** (`shared/components/island-menu/`) — emits `island-mode-change` with `{ mode: 'apariencia' }`. Currently only logged.
- **Admin SPA** (`admin/user/ivangonzalez/index.html`) — mounts the island, loads one theme via binder, has `#card` as the hydration target.
- **Public profile** (`user/ivangonzalez/index.html`) — similar but simpler, no editing.
- **Data model** (`data/card-data.json`) — has `config` block but no theme selection yet.

### Gaps
- Zero JS/CSS for the carousel itself.
- No mapping between `data-name` values and real template files.
- No reaction to `island-mode-change` for "apariencia".
- Theme choice is not persisted or driven by data.
- No visual previews inside the `.card` elements.

---

## Proposed Architecture

### 1. Component Placement (Recommended)
Create a **reusable, framework-agnostic module**:
```
shared/components/card-carousel/
├── card-carousel.js     (core logic + Web Component or init function)
├── card-carousel.css    (horizontal scroll, bullets, transitions, active states)
└── README.md
```

Alternative (faster for v1): embed the logic directly in `admin/user/ivangonzalez/index.html` as a self-contained script + styles, then extract later.

**Recommendation**: Build it as a proper shared component from the start (matches the quality bar of island-menu and binder).

### 2. Data Model Extension
Add to `card-data.json` (and remote source):
```json
"config": {
  "selectedTheme": "aurora",
  ...
}
```
Use the existing `ValueResolver` + binder pattern. The carousel reads/writes this value (or dispatches events that the host page can persist).

### 3. Theme Registry (single source of truth)
Create `data/themes.json` (or a JS module export) with entries like:
```js
{
  "id": "aurora",
  "label": "Aurora",
  "templateUrl": "/templates/cards/aurora.html",
  "previewColor": "#0e0b1a",
  "previewStyle": "..."   // optional inline styles for mini preview
}
```

Map the current skeleton names:
- `aurora` → exists
- `zen`, `tropico`, `obsidiana`, `brutal`, `deco` → exist
- `forest-pattern`, `neon`, `gold` → either rename in skeleton or create stub themes (or treat as future placeholders)

### 4. Carousel Behavior
- **Render**: JS reads the list of themes (from registry or from DOM `data-name` attributes as fallback) and:
  - Ensures each `.card` has a visual preview (CSS background, gradient, or tiny static representation).
  - Dynamically populates `.bullet-active-indicator` with `<button class="bullet" data-name="...">`.
- **Selection**:
  - Click on `.card` or bullet → set `data-card-selected`, add/remove `active` classes.
  - Dispatch a custom event: `card-theme-selected` with `{ themeId, templateUrl }`.
- **Live Preview Swap** (admin only):
  - Host page listens to the event and calls:
    ```js
    binder.loadTemplateAndRun(`/templates/cards/${themeId}.html`);
    ```
- **Persistence** (optional in first pass):
  - On select, update `config.selectedTheme` in a local copy of data and optionally POST to remote.
- **Navigation**:
  - Horizontal scroll (native + momentum).
  - Optional: previous/next arrows or keyboard arrows when focused.
  - Bullets are clickable and also act as progress indicator.

### 5. Integration with Island Menu
In `admin/user/ivangonzalez/index.html` (or a new appearance panel):
- On `island-mode-change` with `mode === 'apariencia'`:
  - Show / mount the carousel (could be injected into a dedicated panel or as an overlay bottom sheet).
  - Hide or collapse the normal card preview while in appearance mode (or keep both side-by-side on desktop).
- On `island-status-change` → 'idle' or check: hide carousel, restore normal view.

### 6. Styling Approach
- Self-contained CSS (no global pollution).
- Modern, premium feel consistent with island-menu (glassmorphism / subtle borders, spring transitions).
- `.card` previews should be small (≈ 80–110px wide) but instantly recognizable as the real theme.
- Active state: scale + ring or colored underline + sync with bullet.
- Bullet indicator: small dots, active dot larger + colored.

Options for previews inside each carousel `.card`:
A. Pure CSS mock (fastest, no extra assets).
B. Scaled-down live template (complex, risk of hydration conflicts).
C. Static screenshot images (best visual fidelity, but maintenance cost).
D. Hybrid: CSS hero + small representative elements.

**Recommended for v1**: Hybrid CSS + representative color + key motif (e.g., aurora has gradient + avatar silhouette).

## Implementation Phases (Recommended Order)

### Phase 0 – Preparation (no breaking changes)
- Extend `card-data.json` with `"config": { "selectedTheme": "aurora" }`.
- Create `data/themes.json` with the 6 existing themes + the 3 placeholder names from the skeleton (map placeholders to existing or mark as "coming soon").
- Add basic CSS variables for carousel in a new file or inside admin styles.

### Phase 1 – Core Carousel Component
- Create `shared/components/card-carousel/card-carousel.js`
  - Export `initCardCarousel(root, options)` and/or a Web Component `<card-carousel>`.
  - Accept either a list of themes or auto-discover from child `.card[data-name]`.
  - Implement bullet generation, click handling, `data-card-selected` sync, custom event dispatch.
- Create `shared/components/card-carousel/card-carousel.css` (importable).
- Make `card-carousel.html` the living demo / test harness (replace its static content with the real component).

### Phase 2 – Visual Previews
- Design and implement mini previews for each of the 6 real themes.
- Ensure previews are static and fast (no heavy DOM inside the tiny cards).

### Phase 3 – Host Integration (Admin SPA)
- Listen to `island-mode-change`.
- When "apariencia" is active: mount the carousel (either replace `#card` content temporarily or show in a dedicated appearance panel).
- On theme selection: call `binder.loadTemplateAndRun(...)` with the chosen template.
- Persist selection back into the local data object (and optionally trigger a save).

### Phase 4 – Polish & Accessibility
- Keyboard navigation (arrows, home/end, enter to select).
- ARIA roles (`tablist` / `tab` pattern or `listbox`).
- Reduced motion respect.
- Touch-friendly scroll snapping.
- Loading / error states if template fetch fails.

### Phase 5 – Public Profile (read-only) – Nice to have
- If `config.selectedTheme` is present, the public profile can render the chosen theme without the carousel.

### Phase 6 – Persistence & Remote Sync (later)
- When user confirms selection (island check button), push updated `config.selectedTheme` to the backend.
- Re-hydrate on load from remote data.

## Risks & Mitigations

- **Template hydration conflicts** when swapping themes rapidly → Mitigate by clearing the container before `loadTemplateAndRun`.
- **Preview maintenance** → Keep previews as pure CSS or very small static assets.
- **Island mode UX** → "Apariencia" must feel like a temporary editor mode, not a permanent state (use `setStatus('edit')` + clear on check).
- **Data model drift** between local JSON and remote → Binder already has fallback; document the new `config.selectedTheme` field clearly.

## Implementation Decisions (Made by Kilo during execution)

All 7 open questions resolved per plan recommendations + project conventions (no blocking user input required; efficient forward progress):

1. **Component vs inline**: Built as proper reusable shared component (`shared/components/card-carousel/`) from the start. Matches island-menu + binder quality bar.
2. **Preview strategy**: Hybrid CSS (gradients + ::after motifs per data-name) in card-carousel.css. Fast, no assets, recognizable. Enhanced in Phase 2.
3. **Carousel placement in "apariencia"**: Dedicated `#appearance-panel` (outside `#card` binder target) injected inside `.main-content` after the live preview. Kept visible alongside card (per handover note). Toggled via hidden attr on mode change. Never replaces preview.
4. **Legacy skeleton names**: Registry contains only the 6 real production themes. `card-carousel.html` updated to real ids. Added `legacyAliases` map in themes.json for future-proofing (not actively used in v1 carousel).
5. **Persistence scope**: In-memory + `localStorage` (via existing admin patterns + `APP_THEME_SELECTED`). Survives binder re-hydrations in session. Full remote sync deferred to Phase 6.
6. **Public profile scope**: Admin editing + live preview swap only (Phase 3). Public (`user/...`) untouched; will read `config.selectedTheme` in Phase 5.
7. **Performance / bundle**: Dynamic `import()` of carousel module + CSS auto-injected only on first "apariencia" activation. Zero cost on initial admin load.
