# Plan: Card Carousel 3 — Premium 3D Visual Theme Selector (Merge of v2 Component + card-carousel-2 Prototype)

## Goal
Create **Card Carousel 3** — a high-fidelity, production-grade evolution of the card theme selector that delivers the stunning UX and visual language of `card-carousel-2.html` (3D perspective transforms, IntersectionObserver-driven depth effects, glassmorphism, live personalization affordances, state machine) while preserving the architectural integrity, reusability, and integration constraints of the existing `shared/components/card-carousel/` (v2) component.

When the user activates **"Apariencia"** mode in the island menu inside the admin SPA:
- A visually spectacular horizontal 3D carousel of the 6 official card themes appears.
- Each preview is instantly recognizable as its corresponding real `templates/cards/*.html` theme.
- Selection instantly swaps the live preview card via the existing binder mechanism.
- The experience feels premium, iOS-polished, and consistent with the quality bar of the rest of the NFC admin surface.
- The component remains a reusable, lazy-loadable, vanilla-first module (with optional advanced rendering layer).

This replaces the current lightweight v2 carousel (and its simple CSS motifs) with a "wow" experience without breaking the binder, island events, or admin SPA.

### Gaps & Tensions
- v2 carousel is intentionally lightweight and vanilla — insufficient visual "wow" compared to the user's vision shown in -2.html.
- card-carousel-2.html is a throwaway high-fidelity prototype built for a different domain (lock screens) using a different stack (Tailwind).
- The 6 real card templates are far more complex than simple lock-screen backgrounds; realistic rich previews are non-trivial.
- Hard constraints from previous architecture:
  - Carousel **must** live outside `#card` (binder does full `innerHTML` replacement).
  - Must remain lazy-loadable (performance budget in admin SPA).
  - Must continue to emit the exact `card-theme-selected` contract.
  - Must respect island menu mode/status events.
  - Must not introduce global pollution or break existing vanilla patterns.
- Open questions around fidelity, tech stack, preview strategy, scope of "personalization", and versioning.

## Proposed Architecture

### Core Principle
**"Advanced Rendering Layer" inside a single reusable component family.**

- Keep `shared/components/card-carousel/` as the public API surface.
- Internally support two rendering modes:
  1. **Standard** (current v2 behavior — lightweight, always available).
  2. **Premium / 3D** (the card-carousel-3 experience — loaded on demand when the host requests "advanced" or when a feature flag / option is passed).
- The 3D engine, IntersectionObserver logic, glassmorphism, and optional personalization panel are implemented as an internal "rich renderer" module (or separate file) that is only fetched when the premium experience is requested.
- This preserves the "lightweight by default + pay for premium" contract.

### Recommended File / Module Layout
```
shared/components/card-carousel/
├── card-carousel.js              (public API + mode router)
├── card-carousel.css             (base + tokens; extended for premium)
├── card-carousel-3d.js           (new: 3D scroller, observer, effects — ported/adapted from -2)
├── card-carousel-3d.css          (premium glassmorphism, perspective styles)
├── previews/                     (optional: static assets or generator for rich previews)
│   ├── aurora.svg
│   └── ...
├── themes-bridge.js              (utility to turn real template snippets into safe preview DOM)
└── README.md                     (updated with v3 usage)
```

Alternative (simpler for first cut):
- Keep everything in `card-carousel.js` + one extra CSS file, gated behind an `advanced: true` option.
- Extract the heavy 3D logic only when `options.advanced` or when a specific data-attribute is present.

### Preview Strategy (Critical Decision)
Options ranked by fidelity vs. cost:

**A. High-fidelity static assets (recommended primary path)**
- Author (or generate) one high-quality representative screenshot / SVG per theme (≈ 220–260px wide, matching the 3D card proportions).
- Store in `shared/components/card-carousel/previews/`.
- Pros: pixel-perfect recognition, zero runtime cost, no style leakage.
- Cons: maintenance when themes evolve.

**B. Runtime isolated mini-renders (hybrid)**
- For each theme, fetch a tiny "preview-only" version of the template (or a curated subset of its DOM + critical CSS) into a shadow root or isolated iframe-like container inside the carousel card.
- Pros: always in sync with real templates.
- Cons: complexity, potential perf hit, risk of CSS variable / font conflicts.

**C. Enhanced CSS motifs + key signatures (current v2 + polish)**
- Dramatically upgrade the existing gradient + ::after system with more sophisticated layered effects that evoke each theme (e.g., aurora mesh noise, brutalist concrete texture, deco gold line patterns).
- Pros: zero new assets, fast.
- Cons: may never reach the "instantly recognizable" bar of real screenshots.

**Recommended for v3**: Start with **A (static assets)** for the 6 themes as the default premium view, with an opt-in path toward **B** later if desired. Provide a clear extension point (`previewRenderMode`).

### Personalization Scope
In the lock-screen prototype the panel mutates font/color of the clock in real time.

For NFC card themes the analogous question is:
- Should the carousel 3 support live "tinting" or "accent override" that is then passed to the live binder swap (via extra detail in the event)?
- Or keep the carousel strictly as a **theme selector** (the 6 canonical experiences) and move any deeper customization to a future "Editar" sub-mode?

**Strong recommendation in plan**: For v3, keep the carousel as pure theme selection. The personalization panel can be implemented later as a separate "theme overrides" surface once the 6 base themes are solid. This keeps scope manageable.

### Integration Points (Must Preserve)
- Same `initCardCarousel(root, { advanced: true, ... })` or `data-carousel-advanced` attribute.
- Same `card-theme-selected` event payload (plus optional `overrides` in future).
- Same lazy import pattern in admin.
- Same `#appearance-panel` container.
- Must continue to work with the existing 6 `templates/cards/*.html` files via `loadTemplateAndRun`.

### Styling & Consistency
- Adopt the island-menu design tokens where possible (`--island-*`).
- Premium 3D layer can introduce its own higher-contrast glassmorphism tokens (inspired by -2) but scoped under `.card-carousel--premium` or a data attribute.
- Avoid shipping Tailwind into the shared component. Either:
  - Pure vanilla CSS port of the glass + 3D effects, or
  - Optional runtime injection of a tiny premium stylesheet (no CDN).

## Risks & Mitigations

- **Visual regression vs. real templates** — Previews will slowly drift if themes are updated frequently. Mitigation: document a clear "preview asset update checklist" and consider a later runtime-isolation path.
- **Bundle size & first-load perf in admin** — The 3D + glass layer + assets could add noticeable weight. Mitigation: strict lazy loading + tree-shaking + optional feature flag.
- **Cross-browser 3D & scroll behavior** — IntersectionObserver + perspective transforms have subtle differences on Safari/iOS. Mitigation: extensive manual testing + feature detection.
- **Style leakage from card templates** — If we ever move toward runtime previews, the heavy `<style>` blocks inside each theme could pollute the carousel. Mitigation: prefer static assets for v3.
- **Maintenance burden** — Two rendering modes inside one component can become confusing. Mitigation: excellent documentation + clear internal module separation.
